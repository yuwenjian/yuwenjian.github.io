1.1 企点接待

l  实现路径，规则

¡  从专业的角度出发，分析问题，并及时解决用户的问题，减少用户等待的时间，以及用最好的态度对待用户，让用户在使用我们的产品时得到最大的满足，有很好的服务体验（SDK企点接待规范&质检规则）

 

l  目标结果考核

¡  1.周评分：根据企点接待客户规范进行评分--商务子骞进行每周考核评分

¡  2.说明：本项为本职工作，不做加分项设定

¡  3.扣分项

n  考核总分平均分超过93分，并且无事故和投诉者则此项满分，85-90，扣2分，80-85，扣4分等

n  有客户投诉，并且由上级认证生效，则按规则和投诉问题的严重性扣1-3分不等

 

1.2 微信群接待

l  实现路径及规则：

¡  1.响应：微信群上班期间5分钟内进行首次响应，休假时间20分钟内进行响应

¡  2.反馈：微信群反馈客户问题，不考虑时间和原因，必须给客户一个解决方案和解决时间，并且征询客户同意

¡  3.话术：微信群客户回复语气，必须保障和蔼，不能进行讽刺客户，如有讽刺，辱骂客户情况，直接解雇处理

¡  4.其他

n  推进问题解决

n  专业性

n  有无重大失误等

n  按照接入规则来对接客户

l  目标结果及考核

¡  加分项：（推动客户的接入，客户的扩展接入等等）

n  1.为维护客户，推进日活增长做出相应贡献

¨  推进内部问题处理，解决并节约客户测试时间周期，加快推进SDK接入带来相应日活增长

¨  协助推进客户使用其他SDK对客户带来的优势，并且发掘竞品SDK的不足来增强我方SDK的使用可能性

¨  甚至能拉进客户其他App的接入我方产品等等

¨  与客户保持粘性，与客户培养相应感情，在策略动作之前提前预知信息

¨  与客户保持粘性，能够推动其他SDK的接入，带来SDK的粘性增长

¨  日常与客户单独交流后，促进客户接入Mob相关SDK，并且后续拉商务群持续跟进，每个加3分

¡  扣分或者惩罚

n  1.微信群（且是等级客户）回复上，存在问题，必须及时纠正，如果后续没有进行纠正，且导致客户弃用，直接评定C

n  2.电话响应规则：上班正常电话接听即可，如果值班人员休息日响应时，未接听，并且发现后未进行回复，一月3次则季度评定扣5分，如有未及时响应跟踪，导致等级S客户弃用，则季度评定直接为C

n  3.客户电话故意不接听两次，解雇处理

n  4.漏接电话，并发现后未及时回复，季度评定C，两次及以上解雇处理

n  5.sla 上班5分钟响应，下班20分

1.3 KA/kB 日常维护

l  实现路径

¡  每天后台查看KA数据情况（SDK等级客户维护规范及质检）

目标结果及考核

¡  惩罚规则：根据汇总情况，每月出现3次异常及时警告--出现第三次警告，季度评定扣5分；

n  1.根据汇总记录情况，2个工作日内如发现异常未汇报给芳芳/守望者，发现2例未汇报情况

¨  如发现2例未汇报-但是实际未弃用情况-季度评定扣5分

¨  如发现1例未汇报-并且有弃用app情况-客户是S级别的季度评定直接为C

¨  如发现两例未汇报-并且有弃用app情况，客户是S级别，直接去劝离团队

n  2.根据记录规则抽查记录频次及内容

¨  如发现记录频次不足，发现一次及时警告，出现两次季度评定扣5分

n  3.如发现记录问题，并且3个工作日弃用app未上报，客户是S级别的季度评定直接为C；

n  4.如发现1例未汇报-并且有弃用app情况-客户是S级别的季度评定直接为C

n  5.如发现两例未汇报-并且有弃用app情况，客户是S级别，直接解雇处理

l 

¡  加分规则：（提高技术支持的积极性，主观能动性，能更好的为业务部门的整体KPI做出贡献）

n  1.观察App日活情况，给出合理性建议，能推动App在Mob侧日活增加（egg：从B级别?CA级别(多渠道接入上线推进等)），则看实际情况评可给技术支持季度评定加分，额度大的话可进行提报A或者S

n  2.观察App使用情况，推动商务解决客户使用不合理情况，长期跟踪，并最终解决问题，让Mob侧数据正常，则相应季度评定加3分

n  3.如果能主动对接客户，了解客户异常，并且能提前预警商务，PM，等，季度评定加3分，如果能反馈及时，成功挽回，则加6分

n  4.日活增加,B级升到A级等情况，视情况增加分数，日活下降来观察给出合理了建议，记录下降的可能性原因，如有记录，并且节省其他部门人员时间，则相应加分

n  5.单独观察到客户使用SDK，并且被PM，商务等所遗漏，及时查看App使用情况，反馈商务跟踪，增加客户粘性，每次加2分

¡  1.季度末能承接最低一个插件的维护更新，从研发手上承接过来进行后续的维护更新

¡  2.能够熟悉了解研发的研发规范规则，并且按照要求开发SDK插件

¡  3.维护插件速度，两周内更新完成

¡  4.说明：本季度先尝试承接，针对质量线做大无重大bug，下季度重点考核技术规范等

 

l  （满分10分，可负分）

¡  1.不按照模板进行提交一次扣1分

¡  2.两周内未推进更新完成，并且没有备注原因的扣1分

¡  3.测试质量，如上线后回归完之后，客户测试出现崩溃现象，则扣5分

¡  4.文档质量，如上线后回归完之后，客户测试出现主要功能无法使用的情况，则扣3分

¡  5.发现问题及时打回上一流程，并添加相关备注，并且进行延期上线调整-记录延期理由等，如未记录则扣3分

¡  6.文档被客户诟病则扣2分

l  加分项

¡  在一周内推进插件更新完成，则增加1分

¡  文档内容实际被客户表扬等，则增加2分

¡  测试如果多次测试出插件bug，并且及时反馈研发流程解决问题，则增加2分

 

1.6 推动/跟进问题

l  实现路径

¡  1.客户反馈bug，进行及时测试

¡  2.测试后确定问题是否可重现，然后反馈研发，说清楚是否可重现等，并提jira流程

¡  3.提交的jira，必须进行记录到confluence空间，并且每周进行汇总汇报等

¡  4.提jira流程，如是客户反馈，我们返回给客户一个工单id，后续客户咨询问题拿工单id来进行咨询即可

¡  5.提交jira后，寻求PM/主程/研发给出排期，如有排期第一时间反馈给客户，与工单id

¡  6.定期跟踪jira，知道解决，解决后及时反馈给相应客户，并且在jira中形成结果备注

¡  7.针对上游客户对接，比如push的厂商，秒验的上游运营商，进行入群等观测，如果上游出现问题及时跟进，组织话术等

¡  8.常见问题汇总，形成沉淀，并且提供优化建议或者方案，最终将客户咨询的重复常见问题降低，进而提升工作效率

 

 

l  目标结果及考核

¡  扣分项：满分10分（可负分）

n  1.工单id规则：提交jira生成的链接工单id后缀。eg：SDKCP-213

n  2.客户反馈问题，没有提jira，发现一次扣2分

n  3.jira没有记录到每周记录表中，发现一次扣2分

n  4.提的jira没有定期跟踪，最后不了了之，发现一次扣2分

n  5.提交的jira解决掉后，没有及时反馈给客户，并且备注到jira中，发现一次扣2分

n  6.上游出现重大问题，及时跟进处理，如未有跟进，并且客户主动反馈的话，则一次扣2分

n  7.常见问题汇总，每两周汇总，如果没有汇总上报，则扣分

¡  加分项：

n  1.重大客户问题全程跟踪，并且稳重大客户情绪，不仅限于技术问题，推动PM,产品,技术等快速解决掉问题，并给大客户及时反馈，并做到，高效，质量，结果三重考评均优秀，则加5分

n  2.重大问题预警，发现客户问题，并且第一时间通过敏感和经验预判问题，迅速做出预警，判断问题等级，最终迅速解决问题，并将风险降到最低等，则加5分

n  3.问题汇总跟踪：强调自觉，针对SDK问题，进行汇总分类，做到优化流程，定期推进问题解决，则考量做的程度2-5分不等。

n  4.上游跟进良好，处理问题速度较快等，并且能够跟上游粘性关系处理妥当，能优先处理Mob相关问题则加分。

n  5.常见问题如果汇总的较好，并且提出优化建议则加分

邮件需要提供的信息如下：

1、变更账号原因说明

2、原账号以及变更为哪个账号

3、提供mob开发者后台“账户资料”页面的截图

4、提供实名认证信息

个人认证用户：提供姓名和身份证号；

公司认证用户：提供公司名称和营业执照编号

申请邮件发送至Mob官方邮箱support@mob.com

 

邮件申请好之后联系―明世隐查看邮件处理

6.2 注销账号

（目前不支持）预计下季度开发改功能

6.3 短信签名模板审核

默认模板：【掌淘科技】： “应用名称”的验证码为：XXXXXX  

注：我们要自定义的就是【】里面的部分，个人用户没有营业执照不支持申请自定义，（个体工商户执照也可以），未申请的用户使用默认签名【掌淘科技】，从而也无法继续下一步申请自定义模板；

 

申请规则：

（1）     一般申请公司简称或者APP名称，审核通过即可申请；

（2）     必须提交营业执照；

（3）     只能3-8个字，不支持全英文，且一个英文字母算一个字，不能添加特殊字符，【】会自动添加；

（4）     申请签名为公司名称/简称，只需上传营业执照即可；

（5）     如申请签名为应用名称，除了要上传营业执照外，另需提供签名的商标证明（商标申请回执）软件著作权证明或者可以看出签名与公司从属关系的应用市场上线截图；

审核时间：正常一个工作日，工作日下午四点之前上传的且符合申请规则的当天审核完成，四点之后上传的正常第二天审核（重点客户可加急处理），休息日不做审核；

签名支持重新申请，通过后的签名，后台点击重新申请按要求换取新的自定义签名提交申请即可重新申请，线上应用不建议重新申请；

6.4 秒验审核

如何申请：后台接入秒验产品后，进入秒验-概况-点击立即审核，填写应用信息，提交审核即可。（包名，MD5必须填写正确）

 

审核流程（内部）：

秒验必须需配置三网（移动，联通，电信）运营商参数，才可获取到本机号码，进行一键登录。

步骤：后台看到用户提交的应用信息，将应用信息(包名，MD5）分别提交至三网运营商，运营商审核通过后，获取到参数（id跟secret等）,再进行后台填写，即审核通过

 

审核时间：1-2个工作日

工作日当天下14点30前提交的秒验，当天18点左右可以审核通过，14点30之后提交的需要等到第二天。周末节假日不审核，需等到工作日。（目前的审核时间是这样，根据运营商统一审核时间为准，特殊情况除外，运营商经常出问题）

 

回复客户统一话术：秒验需提交至三网运营商审核，所以审核时间比较慢，一般是1-2个工作日，还请耐心等待下哦

6.5 企业认证

如何认证：大陆客户直接在mob开发者后台申请认证

需提供的信息：有效企业编码（公司）、身份证号码（个人）提交等待审核

审核时间：后台系统调用接口审核，一般是添加好之后即可审核完成，如显示审核失败，可点击转人工审核，另需上传营业执照（企业）、手持身份证照片（个人），人工处理的需要一个工作日完成审核，客户提供开发者账号，可找客服帮忙加急处理~

港澳台及国外开发者认证单独处理：

注：客户同样需要在后台上传好对应的资料，营业执照（企业）、手持身份证照片（个人）

另需要用户提供以下信息：

企业用户

1.哪个国家或地区

2.公司名称

3.公司具体业务

4.应用名称

5.应用的用户量有多少，是面向哪里的用户

6.会用到我们哪个SDK

7.需要审核账号

8.申请人身份证照片

 

个人用户

1.哪个国家

3.应用名称以及应用涉及哪方面

4.体量多大，会用到我们哪个SDK

 5.申请人身份证照片

 

以上信息跟客户确认后直接发给：小白  等审核结果即可

 

审核时间：正常当天可以处理完成（主要看领导）

6.6 发票

如何申请：

1）后台右上角点击账户余额-财务管理-发票明细  选择对应充值订单即可申请（针对后台支付宝充值客户），申请成功之后会显示待审核；

2）对公付款的，后台查看对应商务，找对应商务协助充值申请；

 

开票周期：一般1周左右  （后台发票页面状态显示已审核未收到票的可以让用户提供开票抬头发小白查询）

 

注：申请即可开票不需要等消耗完
